# Implementation Plan: Authoring UI Layout Fidelity Fixes

**Branch**: `005-authoring-ui-layout-fixes` | **Date**: 2026-06-03 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/005-authoring-ui-layout-fixes/spec.md`

## Summary

The authoring UI renders inside a VS Code webview. Three layout defects arise because the Angular app's CSS was authored for a standard browser environment and has never been adapted for the VS Code webview sandbox:

1. **Button-nav top offset**: `.button-nav` has `top: 64px` — correct for browser chrome (tabs + URL bar), wrong for VS Code webview where the panel starts at `top: 0`.
2. **Edit column right-edge gap**: `modelsConceptsView.html` wraps everything in a `pull-right` float + Bootstrap `col-*` dynamic classes. The VS Code webview body has 8px default padding and no explicit `box-sizing: border-box` reset, causing a right-side gap.
3. **Diagram toggle discoverability**: The "Hide all concept models" button already exists in the fixed button-nav (`edit-no-model` view). It is functionally correct but visually buried in a narrow 50px strip. No additional toggle logic is needed — only layout bug fixes that make this path work correctly in webview.

All fixes are confined to a new webview override stylesheet (`app/styles/vscode-overrides.scss`) loaded conditionally when the app runs inside a VS Code webview. No Angular controllers, templates, or application logic need to change.
